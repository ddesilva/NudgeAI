Architecture — companion window + MCP-bridged agent

The NudgeAI process gets a new Session Workspace window. The agent the user picks (Claude Code, Codex, aider…) runs as a normal subprocess inside an embedded terminal in that window, and talks to NudgeAI over an MCP server hosted in-process. Existing menu-bar capture and review flows are untouched.

Process & data flow
NudgeAI.app (process)
Menu bar + region overlay
⌘⇧N quick capture · Browse Sessions · Settings — unchanged
NEW: Session Workspace window
· tab strip (one tab per loop session)
· split pane: terminal ↔ render pane
· WKWebView (plan HTML or app preview)
· SwiftTerm pty (hosts user's chosen agent)
NEW: In-process MCP server
local unix socket per session · tools: render_plan, open_preview, capture_region, await_user_feedback, get_selection, …
Agent (subprocess in the embedded pty)
User's choice: claude-code / codex / aider / …
launched with NUDGEAI_MCP_SOCKET=… env var · per-agent preset configs ship in app bundle
owns: LLM calls, codegen
tool calls go to: NudgeAI MCP + filesystem + others
NudgeAI never sees the user's API key
MCP over local socket (stdio fallback)
render_plan(html) → preview pane updates · open_preview(url) → WKWebView loads · user clicks element → get_selection() returns crop+selector+html · capture_region() → existing overlay flow inside a session
Persistence: sessions stored under ~/NudgeAISessions/loop-<timestamp>-<slug>/ — extends today's session format with plan.md, plan.html, verify-feedback.jsonl, terminal-transcript.txt. Browseable from existing library window.

Key idea: NudgeAI is "agent-agnostic" because it only ever speaks MCP. Swap claude-code for codex and the workspace UI doesn't notice.
